Training and Inference Workloads

From batch training to real-time serving: how workload shape determines bottlenecks, metrics, scheduler behavior, and principal-level system design tradeoffs.

Current as of 2026 edition

6.0 Chapter Manifesto — Why This Chapter Exists

The first five chapters built the foundation: roofline thinking, transformer arithmetic, GPU architecture, HBM behavior, and the power and thermal limits of AI infrastructure. Chapter 6 changes the unit of analysis from hardware to workload.

A GPU cluster does not run “AI” in the abstract. It runs a workload mode: a multi-day training job, an offline inference batch, a batch API service, or an interactive streaming service. The same model and the same GPU can behave completely differently in each mode.

Principal question

Given a workload, what is the dominant objective, bottleneck, memory pressure, communication pattern, scheduler behavior, and metric that determines whether the system is healthy?

6.1 Workload Taxonomy: Four Modes, Four Objective Functions

The most useful first classification is not model type. It is how much the system can control the batch and how much the user cares about latency.

6.2 Batch Training Pipeline: The Anatomy of a Step

A training job turns data into updated model weights. One step appears simple in Python, but at scale it becomes a pipeline of CPU work, GPU kernels, memory allocation, network collectives, and storage writes.

# Representative PyTorch-style training step [REPRESENTATIVE]
for batch in dataloader:
    batch = move_to_gpu(batch)
    optimizer.zero_grad(set_to_none=True)
    outputs = model(batch.inputs)
    loss = criterion(outputs, batch.labels)
    loss.backward()
    optimizer.step()

    if step % checkpoint_interval == 0:
        save_checkpoint(model, optimizer, step)

6.3 Training Metrics: Step Time, Tokens/sec/GPU, MFU, and Stall Time

step_time = data_time + forward_time + backward_time + communication_time + optimizer_time + checkpoint_overhead
tokens_per_second_per_GPU = useful_training_tokens / (step_time_seconds × GPU_count)
MFU = achieved_model_FLOPs_per_second / theoretical_peak_FLOPs_per_second

For language model training, use useful non-padding tokens. Padding-heavy workloads can make a system look healthier than it is. GPU utilization tells you whether the GPU is busy; MFU tells you whether the busy time is doing useful model math.

6.4 Training Memory: Weights, Gradients, Activations, Optimizer State

Training requires more memory than inference because backward propagation needs gradients, optimizer state, and either stored or recomputed activations.

Checkpointing is a tradeoff

Activation checkpointing lowers activation memory by recomputing selected activations during backward. It can enable a larger batch or model, but increases step time.

6.5 Distributed Training Communication: Why Scaling Is Not Free

In data parallel training, each replica computes gradients for a different microbatch, then synchronizes gradients so all replicas apply the same update.

local forward/backward → gradients → AllReduce or reduce-scatter → optimizer update

Scaling efficiency is controlled by the ratio of communication time to useful compute time. If communication time is hidden behind backward compute, scaling is efficient. If exposed on the critical path, adding GPUs gives diminishing returns.

6.6 Data Loading and Checkpointing: The Non-GPU Bottlenecks

Production training traces often show that the job is only as fast as the slowest pipeline stage. A GPU can starve because data arrives late or checkpointing pauses too often.

checkpoint_overhead_fraction = checkpoint_pause_time / checkpoint_interval_wall_time

If checkpointing happens too often, the cluster loses expensive GPU-hours. If it happens too rarely, failures lose too much work. This is a reliability-throughput tradeoff.

6.10 Chunked Prefill: Protecting Decode from Long Prompts

A long prompt can monopolize compute and delay active decode requests. Chunked prefill breaks a long prefill into smaller chunks and interleaves decode iterations between chunks.

6.11 Disaggregated Prefill/Decode Serving

At large scale, prefill and decode can have sufficiently different resource profiles that separating them becomes attractive.

6.12 KV Cache as the Serving Resource Bridge

KV_cache_bytes ∝ active_sequences × sequence_length × layers × KV_heads × head_dim × bytes_per_element

KV cache grows with active concurrency and context length. A runtime can have available compute but insufficient KV memory for new requests.

KV guardrail wording

A KV-utilization alert threshold around 80–85% can be useful, but it must be tuned to the model, runtime, page size, preemption behavior, traffic distribution, and SLO. Do not treat 82% as universal.

6.14 Throughput-Latency Knee and Multi-Tenant Serving

Interactive serving should not blindly maximize average GPU utilization. Queueing systems have a knee: as request rate approaches service capacity, latency increases nonlinearly.

6.16 Principal-Level Workload Discussion

Training design answer

Start with objective, tokens, sequence length, global batch, memory budget, parallelism, communication, data pipeline, checkpoint/recovery, and measurement. Do not jump straight to NCCL or a framework name.

Serving design answer

Start with TTFT/TPOT P99 SLO, traffic shape, prefill/decode split, KV budget, scheduler behavior, runtime choice, routing, autoscaling, and measurement. Do not say only “use vLLM because it is fast.”

6.17 Anti-Patterns

  1. Treating training and serving as the same workload.
  2. Reporting average latency only.
  3. Maximizing utilization without SLO headroom.
  4. Ignoring useful tokens and padding waste.
  5. Blaming kernels before checking input pipeline stalls.
  6. Using fixed-batch intuition for interactive serving.
  7. Forgetting optimizer state and checkpoint cost.
  8. Universalizing framework benchmark claims without trace replay.

6.18 Key Takeaways

  1. Training, offline inference, batch API serving, and interactive serving have different objective functions.
  2. Training bottlenecks include compute, activation memory, optimizer state, data loading, communication, and checkpoint I/O.
  3. LLM inference has two phases: prefill and decode.
  4. TTFT is shaped by queueing and prefill; TPOT is shaped by decode efficiency, scheduler policy, HBM/KV pressure, and batch behavior.
  5. Percentile latency matters more than average latency for interactive serving.
  6. Continuous batching schedules at token-iteration boundaries.
  7. KV cache is a first-class serving resource.
  8. Principal-level workload analysis begins with objective, bottleneck, metric, and validation experiment.
